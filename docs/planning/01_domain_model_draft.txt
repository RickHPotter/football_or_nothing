Football or Nothing - Domain Model Draft
========================================

Status
------
Draft. This is a planning document, not a migration contract yet.


Authentication and Player Identity
----------------------------------
User
- Rails authentication identity.
- Use the default Rails authentication generator introduced in recent Rails.
- Owns one or more manager careers.
- Should not directly represent an in-game football professional.
- Should never represent a club owner in the core gameplay loop.

Manager
- In-game character controlled by a user.
- Belongs to a user.
- Can have contracts with clubs over time.
- Should have reputation, nationality, age/birthdate, career status, and
  possibly RPG-style attributes.
- Should not be permanently tied to one club.
- Starts with low reputation and should not naturally qualify for elite
  international jobs at the beginning of a career.
- Current implementation uses reputation to limit available jobs and applies
  reputation gains from completed seasons.


Football Geography
------------------
Country
- Represents a football nation.
- Used by athletes, managers, clubs, stadiums, and tournaments.
- Can later hold federation settings, calendar style, coefficients, or ranking.

Club
- Represents a football club or possibly a national team later.
- Belongs to a country.
- Has a name, short name, reputation, founding year, finances, and status.
- Has a flag/boolean for international scope if needed, but this needs naming
  care. "international" could mean global club, national team, or tournament
  scope.
- Has many athletes through contracts or squad memberships.
- Has many stadium relationships over time if stadiums can change.
- Should be generated for the first versions, with real data import treated as
  a later input path.
- Current implementation applies champion reputation growth at season
  finalization.

ClubFinance
- Financial state for a club.
- Can start as fields on Club if simple, but will likely deserve its own model
  once history, budgets, or accounting periods matter.
- Tracks cash balance, wage budget, transfer budget, debt, revenue, expenses,
  prize money, sponsorship income, and stadium income.

Stadium
- Represents a venue.
- Belongs to a country and optionally a city.
- Can belong to a club as current home stadium.
- Holds capacity, pitch quality, rent/ownership data, and expansion potential.


People and Squads
-----------------
Athlete
- Represents a player.
- Belongs to a country as nationality.
- Has identity data: name, birthdate/age, preferred foot, height, weight.
- Has football data: position, attributes, potential, reputation, morale,
  condition, injury status, and retirement status.
- Should be connected to clubs through contracts, not only club_id, so history
  is preserved.
- Current implementation applies simple age-based ability progression/decline
  and morale/condition updates when a tournament edition is finalized.
- Current implementation also tracks injury and suspension dates for squad
  availability.

AthleteAttributeSet
- Attribute container for an athlete.
- Can start as columns on Athlete if that is simpler, but separate modeling may
  help with historical snapshots, training, scouting uncertainty, and generated
  data.
- Target around 20 core attributes.
- Candidate attributes:
  finishing, long_shots, passing, crossing, dribbling, technique, first_touch,
  tackling, marking, positioning, heading, pace, acceleration, stamina,
  strength, jumping, decisions, composure, teamwork, work_rate.

Position
- Represents broad football positions such as goalkeeper, center back, full
  back, defensive midfielder, central midfielder, attacking midfielder, winger,
  striker.
- Can be enum-backed at first if the list is stable enough.

TacticalRole
- Role available within one or more positions.
- Examples: poacher, target forward, advanced playmaker, ball-winning
  midfielder, inverted winger, wing back, ball-playing defender, sweeper keeper.
- Roles should derive from positions, but should not be identical to positions.

AthleteContract
- Join/history model between athlete and club.
- Tracks start date, end date, wage, release clause, squad number, status, and
  whether it is current.
- Allows loans, free agents, renewals, and contract history later.

ManagerContract
- Join/history model between manager and club.
- Tracks start date, end date, wage, role, expectations, status, and whether it
  is current.
- Allows manager career history.


Competitions
------------
Tournament
- Represents a competition.
- Belongs to a country when domestic.
- May be international/global when not tied to one country.
- Has format data: league, cup, group stage, knockout, mixed.
- Has season/calendar configuration.
- International competitions should use the same model where possible, with
  scope and participation rules deciding behavior.

TournamentEdition
- One specific season/edition of a tournament.
- Example: National League 2026, Continental Cup 2027.
- Stores status, start date, end date, rules snapshot, and champion.

TournamentParticipation
- Join model between club and tournament edition.
- Tracks club entry, final position, points, record, prize money, and status.
- This is the right place for standings/history data.

Fixture
- A scheduled match.
- Belongs to a tournament edition when competitive.
- Has home club, away club, stadium, scheduled date, status, and result.
- Should support live simulation state once match time is modeled.

MatchEvent
- Historical timeline event for a fixture.
- Current implementation stores goals, major chances, yellow cards, red cards,
  injuries, and substitutions with minute, club, athlete, and description.
- Future event types should include cards, injuries, substitutions, tactical
  changes, and major chances.

MatchResult
- Match outcome and aggregate stats.
- Can be part of Fixture or separate if simulation details become large.
- Stores goals, possession, cards, injuries, attendance, and rating summaries.

MatchStat
- Future per-club fixture statistics model.
- Stores possession, shots, shots on target, fouls, yellow cards, red cards,
  and other aggregate match stats.
- Useful once tactical simulation affects more than scoreline and timeline.

MatchState
- Runtime or persisted state for a match in progress.
- Tracks match minute, clock status, score, current lineups, substitutions,
  tactical choices, incidents, and whether the manager has paused time.
- Important for the Brasfoot-like match flow where time advances and the manager
  can stop play to make changes.
- Current implementation stores minute, clock status, and substitution counts.

Lineup
- Club selection for a fixture.
- Tracks formation, starters, bench, captain, and tactical setup.
- Current implementation stores fixture, club, formation, mentality, and status.

LineupAthlete
- Join model between lineup and athlete.
- Tracks selected position, tactical role, starter/bench status, and
  substitution timing.
- Current implementation supports starter/bench status and substitution timing.


Transfers and Market
--------------------
Transfer
- Historical movement of an athlete between clubs.
- Tracks from club, to club, athlete, transfer date, fee, type, and status.
- Types may include permanent, loan, free transfer, release, youth promotion.
- Current implementation supports completed permanent, free, and loan transfers,
  updates current contracts, and enforces transfer and wage budget checks.

Loan
- Current implementation uses `Transfer` with type = loan and loan end date,
  plus a current loan `AthleteContract`.
- Loan behavior preserves the parent club contract as non-current while
  temporarily making the athlete available to the loan club.
- Loan expiry restores the parent contract and expires the temporary loan
  contract.
- Deferred details include recalls, buy options, and complex wage splits.

TransferOffer
- Future model for negotiations.
- Tracks bidding club, selling club, athlete, fee, wage proposal, status, and
  expiration.
- Current implementation stores pending offers and can finalize acceptable
  permanent, free, and loan offers into completed transfers.

AiTransferPlanner
- Future service concept for transfer-window world movement.
- Should inspect squad needs, budgets, club reputation, and available players
  before creating offers or free-agent signings.
- Should avoid repeated churn for the same athlete within one transfer window.

SquadNeedsAnalyzer
- Future service concept that compares a club's available squad against desired
  position depth.
- Inputs should include injuries, suspensions, loans, contracts, and formation
  preferences.


Training and Development
------------------------
TrainingPlan
- Club-level training configuration controlled by the manager.
- Tracks focus, intensity, active dates, and possibly manager/staff ownership.
- Early focuses: balanced, fitness, attacking, defending, technical, youth
  development.
- Current implementation stores one active plan per club and applies it during
  date advancement.

TrainingResult
- History model for visible development changes.
- Tracks athlete, club, plan, changed attribute, old value, new value,
  condition change, and date.
- Current implementation shows recent results on the club dashboard.


Scouting
--------
ScoutingAssignment
- Represents a club scouting task.
- Tracks target country, position, focus, status, starts_on, and ends_on.
- Should generate or refresh scout reports over time.
- Current implementation processes completed assignments during date
  advancement.

ScoutReport
- Club-specific knowledge about an athlete.
- Tracks observed current ability, observed potential ability, confidence,
  summary, and creation date.
- Allows the transfer market to show uncertain information before a player is
  fully scouted.
- Current implementation stores one report per club/player and uses observed
  ability in the transfer market when available.


Staff
-----
StaffMember
- Non-player football professional.
- Roles include assistant manager, coach, fitness coach, scout, and physio.
- Has reputation and role-relevant attributes.

StaffContract
- Join/history model between staff member and club.
- Tracks wage, start date, end date, current flag, and status.
- Staff should affect training, scouting, recovery, and advice.


Youth Development
-----------------
YouthAcademy
- Club youth setup or club-level academy fields.
- Holds academy quality and possibly annual operating cost.

YouthIntake
- Yearly event that creates academy prospects.
- Tracks club, season year, generated date, and generated athletes.
- Quality should depend on club reputation, country reputation, and academy
  quality, with rare high-potential outliers.


Achievements and History
------------------------
Trophy
- Represents a title won by a club or manager.
- Usually belongs to tournament edition.
- Links winner club, manager, and possibly captain/top scorer later.
- Implemented as the historical title event created when a tournament edition
  is finalized.

ClubTrophy
- Optional join/history model if Trophy represents trophy type instead of event.

ManagerAchievement
- Future RPG/career record.
- Examples: promotion, avoided relegation, won derby, unbeaten run.

TournamentFinalizer
- Service concept that closes a tournament edition when all fixtures are
  completed.
- Assigns champion from standings, marks participations complete, creates
  trophy history, and snapshots club/manager season stats.


Game Time and World State
-------------------------
GameSave or Career
- Top-level saved game/career world.
- Belongs to user.
- Owns the generated universe if multiple saves are allowed.
- Important if users can have multiple parallel worlds.
- Owns generated fictional data until real-data import exists.

Season
- Represents one year/cycle in a career world.
- Used by tournaments, contracts, stats, and aging.
- Current implementation uses tournament editions as the concrete season unit
  and rolls completed editions into the next scheduled edition.

SeasonRollover
- Service concept that starts the next season from a completed tournament
  edition.
- Carries participating clubs forward, creates fresh standings, generates
  fixtures, and preserves previous season history.
- Current implementation is idempotent enough to avoid duplicate next editions,
  participations, and fixtures.

CalendarEvent
- Optional future model for scheduled simulation events.
- Examples: match day, transfer window opening, contract expiration, youth
  intake, season rollover.
- Useful for a time-advancing simulation where the world moves through dates
  rather than only instant screens.


Data Generation and Import
--------------------------
WorldGenerator
- Service concept, not necessarily an Active Record model.
- Generates countries, clubs, stadiums, athletes, contracts, tournaments, and
  starting finances.
- First versions should use fictional data.

DataImport
- Future model or service for importing real football data.
- Should map external records into the same domain models instead of creating a
  parallel real-data schema.
- Import should be idempotent and source-aware, using external source/id fields
  or a dedicated import run table.

DataImportRun
- Optional audit/history model for imports.
- Tracks source, started_at, finished_at, status, counts, and errors.

ExternalIdentity
- Optional normalization model if multiple imported models need source/id
  mapping without adding external fields to every table.


News and World Activity
-----------------------
NewsItem
- Generated world/career event visible to the player.
- Tracks category, title, body, occurred_on, and optional links to club,
  athlete, manager, tournament, fixture, or transfer.
- Categories should include match, transfer, injury, discipline, trophy,
  contract, youth, and world.


Statistics
----------
AthleteSeasonStat
- Per-athlete, per-club, per-season statistics.
- Goals, assists, appearances, cards, injuries, minutes, average rating.
- Current implementation tracks appearances, goals, assists, minutes, yellow
  cards, red cards, injuries, and an average rating placeholder.

ClubSeasonStat
- Per-club, per-season aggregate statistics.
- Wins, draws, losses, goals for/against, finances, reputation change.
- Current implementation snapshots final position, record, points, goal
  difference, and champion flag.

ManagerSeasonStat
- Per-manager, per-season statistics.
- Matches, wins, trophies, reputation change, job changes.
- Current implementation snapshots the club managed, final position, match
  record, trophies, and reputation change placeholder.

Modeling Notes
--------------
Avoid putting current-only foreign keys everywhere too early.

For example, Athlete could have current_club_id later for convenience, but the
source of truth for club history should be AthleteContract or a similar
historical relationship.

Likewise, Manager should not simply have club_id if the career can move between
clubs.

Use explicit history/join models for anything the game needs to remember:
contracts, transfers, tournament entries, trophies, jobs, squad membership, and
statistics.

Do not defer money entirely. If finances are core from the start, the first club
model needs at least enough financial data to make wages and transfer decisions
meaningful.
