Football or Nothing - First Playable Backlog
============================================

Goal
----
Reach a minimal playable football-management loop before expanding the world.


Phase 1 - Foundation
--------------------
Status: mostly implemented.

1. Add default Rails authentication.
2. Create the user-to-manager career concept.
3. Create countries.
4. Create clubs.
5. Create initial club finances.
6. Create stadiums.
7. Create athletes.
8. Create athlete attributes.
9. Create athlete contracts.
10. Generate a tiny fictional world:
   - 1 country
   - 8 clubs
   - 8 stadiums
   - 18 to 25 athletes per club
   - Basic financial profiles for each club
   - 1 manager controlled by the user

Phase 14 - Loans
----------------
Status: implemented.

Goal:
Support temporary player movement without destroying parent-club contract
history.

Scope:
- Add loan support to the transfer flow.
- Track loan start date, end date, wage contribution, and optional loan fee.
- Keep the parent club contract intact while the player is temporarily
  available for the loan club.
- Include loaned-in players in the loan club squad.
- Exclude loaned-out players from the parent club lineup pool.
- Return players to parent clubs when loans expire.

Implemented:
- `Transfer` and `TransferOffer` support `loan` transfer type and loan end date.
- Loan offers can be submitted and finalized through the transfer market.
- Loan completion pauses the parent contract by setting it non-current without
  terminating it.
- Loan completion creates a current loan contract for the destination club and
  links it to the parent contract.
- Current squad and lineup pools naturally include loaned-in athletes and
  exclude loaned-out athletes through current contract membership.
- `LoanExpiryProcessor` returns expired loan athletes to their parent contracts.
- Season rollover expires loans before normal contract expiry.
- Clubs expose helper methods for loaned-in and loaned-out contracts.

UI:
- Transfer market supports buy and loan offers.
- Open offers and recent transfer lists show transfer type and loan end date.

Deferred:
- Loan recalls.
- Buy options.
- Complex wage splits.
- AI loan negotiations.
- Club dashboard labels for loaned-in and loaned-out players.
- Athlete profile loan context.


Phase 15 - AI Squad Movement
----------------------------
Status: planned.

Goal:
Make the world move during transfer windows without relying only on the player.

Scope:
- AI clubs inspect squad needs.
- AI clubs sign free agents when affordable.
- AI clubs submit simple transfer offers.
- AI clubs renew or release expired players conservatively.
- Run AI movement during transfer windows and season rollover.

Likely services:
- `AiTransferPlanner`
- `SquadNeedsAnalyzer`
- `AiContractRenewalProcessor`

Rules:
- Clubs target weak positions first.
- Clubs respect transfer and wage budgets.
- Low-reputation or low-budget clubs prefer free agents.
- High-reputation clubs target higher-ability players.
- Avoid moving the same player repeatedly in one window.

UI:
- Transfer market shows recent AI transfers.
- Career dashboard gets a compact world activity panel.

Deferred:
- Bidding wars.
- Player preferences.
- Agents.
- Advanced squad registration.


Phase 16 - Training
-------------------
Status: planned.

Goal:
Give the manager a recurring development lever for squad growth and condition.

Scope:
- Add club training focus:
  - balanced
  - fitness
  - attacking
  - defending
  - technical
  - youth development
- Add training intensity.
- Apply training effects on date advancement, month advancement, or season
  rollover.
- Training should affect attributes, condition, morale, and injury risk.

Likely models:
- `TrainingPlan`
  - club
  - manager
  - focus
  - intensity
  - active_from
- Optional `TrainingResult`
  - athlete
  - changed attribute
  - old value
  - new value
  - occurred_on

Rules:
- High intensity improves faster but lowers condition.
- Low condition should increase injury risk in later match simulation phases.
- Young high-potential athletes improve more.
- Older athletes maintain or decline.

UI:
- Club dashboard training panel.
- Athlete profile development history.
- Career dashboard monthly training summary.

Deferred:
- Staff modifiers.
- Facilities.
- Individual player training.
- Tactical familiarity.

Phase 23 - Data Import Foundation
---------------------------------
Status: planned.

Goal:
Prepare for importing real football data without corrupting generated careers.

Scope:
- Add import services that map external data into existing domain models.
- Preserve fictional world generation.
- Avoid duplicate countries, clubs, athletes, and contracts.
- Make imports idempotent.

Likely services:
- `DataImport::CountriesImporter`
- `DataImport::ClubsImporter`
- `DataImport::AthletesImporter`
- `DataImport::ContractsImporter`

Likely model additions:
- `external_source`
- `external_id`
- Optional `DataImportRun`

Rules:
- Generated and imported data can coexist.
- External IDs are namespaced by source.
- Import should not overwrite save-specific history.

UI:
- No full UI required at first.
- Start with task/runner-driven import.

Deferred:
- Automated scraping.
- Licensing checks.
- Live data updates.
- Conflict-resolution UI.
